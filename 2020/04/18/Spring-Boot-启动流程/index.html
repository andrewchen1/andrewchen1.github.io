Spring Boot 启动流程

Posted on 2020-04-18 Edited on 2020-04-25 In SpringBoot揭秘

Spring Boot应用打包

在编写spring boot 项目的时候往往会在pom的build 部分内加入

其作用就是重新组织资源 使最终的结果物 无论是jar形式的还是 war形式的 都能 通过java -jar ***.jar(war)的命令启动。

在通过maven 打包的时候

首先按照会正常的流程编译，之后

spring-boot-maven-plugin 插件 会介入进行repackage

根据在pom 中 packaging 指定的类型不同 jar/war 重新组织文件的结构和内容

  1. 如果是war类型的

    对artifacts进行解压后会有如下的文件结构

    META-INF/ MAINFEST.MF

    META-INF/maven/com/example/demo/pom.properties

    META-INF/maven/com/example/demo/pom.xml

    WEB-INF/classes/*

    WEB-INF/lib/*

    org/springframework/boot/loader/*

    其中org/springframework/boot/loader/* 所代表的是下面这个包内的内容

其作用是引导Spring Boot应用加载资源

同时会修改 MAINFEST.MF 的内容 ,值得注意的是Main class 并不是在程序里面写的入口类，而是Spring Boot提供的一个入口类

  1. 如果是jar 类型的

    对jar进行解压后会有如下的文件结构

    META-INF/ MAINFEST.MF

    META-INF/maven/com/example/demo/pom.properties

    META-INF/maven/com/example/demo/pom.xml

    BOOT-INF/classes/*

    BOOT-INF/lib/*

    org/springframework/boot/loader/*

    差异点有

    把WEB-INF 替换为 BOOT-INF

    MAINFEST.MF 中的Main-Class 为org.springframework.boot.loader.JarLauncher

Spring Boot项目启动

首先会加载 main-class 指定类的main方法。然后由其调用start-class 中的main方法

Main-Class部分

ONE

  • Launcher

    这个类的作用是加载程序。JarLauncher 和 WarLauncher 都是继承于ExecutableArchiveLauncher

    在ExecutableArchiveLauncher的构造方法中调用

    Launcher#createArchive

    根据当前class的位置是在文件中还是文件夹中 判断程序jar类型的还是war类型的

    如果是jar类型的 文件组织方式是 /BOOT-INF/lib /BOOT-INF/classes

    如果是war类型的 文件的组织方式是 WEB-INF/lib WEB-INF/classes

    生成对应的 JarFileArchive ExplodedArchive 供后续的文件遍历使用

    父类加载完成 子类的构造函数中（JarLauncher / WarLauncher）中并没有什么内容

    对于explored war 通过遍历root path 下的所有内容 将WEB-INF/classes/和WEB-INF/lib/， WEB-INF/lib-provided/

Start-Class 部分

SpringApplication.run(Application.class, args);
然后
SpringApplication#run(new Class<?>[] {primarySource}, args);
然后在这个方法中会创建一个SpringApplication 的实例

new SpringApplication(primarySources).run();

实际会调用的SpringApplication的构造方法是SpringApplication(ResourceLoader resourceLoader, Class<?> … primarySources)

此时的 ResourceLoader的值为空 primarySources的值是我们在main函数里面提供的**Application.class

这个方法里面比较重要的是

  1. setInitializers 入参是ApplicationContextInitializer.class
  2. setListeners 入参是ApplicationListener.class

主要是从getSpringFactoriesInstances 中去找 ApplicationContextInitializer.class 和ApplicationListener.class的子类

getSpringFactoriesInstances比较像SPI 是Spring私有协议，解析每个jar包里面的META-INF/spring.factories文件。

在项目启动的时候 会把BOOT-INF 里面所有的jar都由ClassLoad 载入一次了

SpringApplication的实例就算构造好了 然后开始执行SpringApplication的Run方法。

  1. 在spring.factories的list里面找SpringApplicationRunListener的子类 然后创建他们的实例。调用他们的starting方法。

  2. 查找并且加载SpringApplicationRunListener的实现类，调用他们的start方法

  3. 创建并且配置将要使用的Environment（Properties和Profile）将启动参数的里面的内

    主要的方法是SpringApplication#prepareEnvironment

主要做的事情是把commandline的参数加入到environment 中。

  1. 启动Application

  2. 创建reporters的实例

  3. 调用SpringApplication#prepareContext

    1. 设置Environment

    2. postProcessApplicationContext

      配置 (自定义的) beanNameGenerator

      配置（自定义的）resourceLoader

      配置（自定义的）classLoader

      配置conversionService

    3. 调用applyInitializer 调用ApplicationContextInitializer#initialize 方法 （ApolloApplicationContextInitializer#initialize 开始在这里发挥作用 拉取所需要的配置放到context的environment 中）

    4. 调用 listeners.contextPrepared(context) listen 是SpringApplicationRunListeners类型

    5. 设置beanFactory 的

      1. springApplicationArguments
      2. printedBanner

    6. 设置BeanDefinitionLoder的信息

      如果SpringApplication#beanNameGenerator 不是null 设置给BeanDefinitionLoder

      如果SpringApplication#resourceLoader 不是null 设置给BeanDefinitionLoder

      如果SpringApplication#environment 不是null 设置给BeanDefinitionLoder

    7. 调用SpringApplicationRunListeners#contextLoaded

  4. refreshContext

    1. prepareRefresh

      web项目会重写这个方法

      AbstractRefreshableWebApplicationContext#initPropertySources 会读取servletContextInitParams和servletConfigInitParams中的内容
